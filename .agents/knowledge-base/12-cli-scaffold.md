## What the Scaffold Seam Is

`netscript init` produces a **self-contained, Aspire-orchestrated project skeleton**. It can start
as an intentionally small shell, or include an example service and database when the relevant flags
are passed. Follow-on capability commands then compose into that project: `db add` provisions
database workspaces, `service add` registers Deno services, and `plugin add` scaffolds or copies
plugin workspaces and updates Aspire wiring.

The scaffold is driven entirely by `appsettings.json`. Every subsequent CLI command (`netscript db`,
`netscript service`, `netscript plugin`) reads that file and mutates or extends the project
accordingly. Commands that change topology also regenerate the TypeScript Aspire helper layer. The
root `generate` command includes `generate plugins`, which discovers installed plugin contributions
and writes per-axis registries under `.netscript/generated/`.

## Project Anatomy

```
<project-name>/
├── appsettings.json          ← master config: DBs, services, apps, tools
├── netscript.config.ts       ← project identity, paths, feature flags
├── deno.json                 ← workspace root (all members listed here)
│
├── aspire/                   ← TypeScript AppHost (Node — isolated from Deno workspace)
│   ├── apphost.ts            ← entry point; calls createNetScriptAppHost()
│   ├── package.json
│   ├── .modules/             ← Aspire TS SDK; populated by `aspire restore`
│   └── .helpers/             ← GENERATED by CLI; never edit by hand
│       ├── index.ts          ← orchestrator: createNetScriptAppHost()
│       ├── configure-dashboard.ts
│       ├── register-infrastructure.ts
│       ├── register-services.ts
│       ├── register-plugins.ts
│       ├── register-background.ts
│       ├── register-apps.ts
│       ├── register-tools.ts
│       ├── db-cli-mode.ts    ← DB operation short-circuit
│       └── _aspire-compat.ts
│
├── database/
│   └── <engine>/             ← one directory per DB engine (postgres / mysql / mssql / sqlite)
│       ├── deno.json         ← DB workspace; owns db:* task aliases
│       ├── mod.ts            ← generated facade: export { db }
│       ├── prisma.config.ts  ← Prisma v7 config with URL normalisation
│       ├── schema/
│       │   ├── schema.prisma ← starter schema; add your models here
│       │   └── plugins/      ← copied plugin schema contributions
│       ├── migrations/       ← created by db init / db migrate; commit to git
│       ├── scripts/
│       │   ├── migrate.ts    ← thin wrapper over @netscript/database/scripts
│       │   ├── generate-zod.ts
│       │   ├── seed.ts       ← replace ExampleRecord upsert with real seeds
│       │   └── fix-zod-imports.ts
│       └── schema/.generated/   ← generated output; do NOT commit
│           ├── client.server.ts ← created by runPatchPrismaClient (all engines)
│           ├── zod/          ← Zod schemas, one file per model
│           └── …
│
├── apps/
│   └── dashboard/            ← Fresh 2 + Preact + Vite frontend
│
├── contracts/                ← oRPC contract stubs (v1); expand per domain entity
│   └── versions/v1/
│
├── services/                 ← created only when --service or service add is used
│
├── plugins/                  ← installed plugin service workspaces
│   └── <plugin>/database/    ← plugin-local schema source when DB-backed
│
├── workers/                  ← background workspace copied by official workers plugin
├── sagas/                    ← background workspace copied by official sagas plugin
├── triggers/                 ← background workspace copied by official triggers plugin
├── .netscript/generated/     ← generated worker/saga/trigger registries
│
└── packages/                 ← local copies of @netscript/* when scaffolded by netscript-dev
```

**Key default**: without `--service`, `services/` is not created and `appsettings.json` ships with
`"Services": {}`. Services are domain-specific; add one with `--service` during init or later with
`netscript service add`.

## Configuration Model

`appsettings.json` is the single config file the TypeScript AppHost reads. Every section maps to a
generated helper file in `.helpers/`.

```jsonc
{
  "NetScript": {
    "Name": "<project-name>",
    "Version": "1.0.0",
    "PrimaryDatabase": "<config-key>", // which DB is primary
    "PrimaryCache": "<config-key>", // which cache is primary

    "Otel": {
      "HttpEndpoint": "http://localhost:4318",
      "Protocol": "http/protobuf"
    },

    "Defaults": {
      "Deno": {
        "Permissions": ["--allow-net", "--allow-env", "--allow-read", "--allow-sys"],
        "WatchMode": false
      }
    },

    "Databases": {
      "<config-key>": {
        "Engine": "Postgres | Mysql | Mssql | Sqlite",
        "Mode": "Container | External", // Container = Docker-managed
        "DatabaseName": "<db-name>",
        "Persistent": true // container volume survives restarts
      }
    },

    "Cache": {}, // Garnet entries

    "Services": {
      "<name>": {
        "Runtime": "deno",
        "Entrypoint": "src/main.ts",
        "Workdir": "services/<name>",
        "Port": 3001,
        "DependsOn": ["<other-service>"] // optional wait chain
      }
    },

    "Plugins": {
      "<name>": {
        "Runtime": "deno",
        "Entrypoint": "services/src/main.ts",
        "Workdir": "plugins/<name>",
        "Port": 8091,
        "RequiresDb": true,
        "RequiresKv": true,
        "PluginReferences": ["<other-plugin>"]
      }
    },

    "BackgroundProcessors": {
      "<name>": {
        "Runtime": "deno",
        "Entrypoint": "bin/combined.ts",
        "Workdir": "<name>",
        "RequiresDb": true,
        "RequiresKv": true
      }
    },

    "Apps": {
      "dashboard": { "Runtime": "deno", "Type": "app", "Port": 8010 }
    },

    "Tools": {
      "prisma-studio": {
        "Enabled": true,
        "TaskName": "db:studio",
        "Database": "<config-key>"
      }
    }
  }
}
```

Mutating `appsettings.json` through CLI capability commands regenerates the affected `.helpers/`
files. If you edit `appsettings.json` manually, run the owning command again (`service generate` for
services) or re-run the scaffold operation that owns that topology. Do not hand-edit `.helpers/`.

## `netscript init` — Command Reference

```bash
netscript init <project-name> [flags]
```

| Flag                    | Purpose                                                                         |
| ----------------------- | ------------------------------------------------------------------------------- |
| `--app-name <name>`     | Frontend application name                                                       |
| `--db <engine>`         | Include a database workspace: `postgres`, `mysql`, `mssql`, `sqlite`, or `none` |
| `--service`             | Scaffold an example oRPC service                                                |
| `--service-name <name>` | Example service name; implies `--service`                                       |
| `--service-port <port>` | Example service port; implies `--service`                                       |
| `--path <dir>`          | Parent directory (default: cwd)                                                 |
| `--editor <editor>`     | Optional editor config: `none`, `zed`, `vscode`                                 |
| `--no-aspire`           | Skip Aspire orchestration                                                       |
| `--legacy-aspire`       | Generate the legacy C# AppHost instead of the TypeScript AppHost                |
| `--ci`                  | Skip interactive prompts; accept all defaults                                   |
| `-y`, `--yes`           | Accept defaults without prompting                                               |
| `--no-git`              | Skip `git init`                                                                 |
| `--force`               | Overwrite if target directory already exists                                    |

`--db` can be omitted to scaffold without a database. A database can always be added later with
`netscript db add <engine>`.

After `init` the Aspire SDK must be downloaded once:

```bash
cd <project-name>/aspire && aspire restore
```

## `netscript db` — Command Family

All `db` commands delegate to a dedicated, isolated Aspire AppHost invocation. They go through the
full Aspire orchestration path so the database container is provisioned and healthy before the
Prisma operation runs.

### Invocation

In local contributor mode (`netscript-dev init` from the monorepo), the CLI package is copied into
scaffolded projects at `packages/cli/`. Run commands directly:

```bash
deno run -A packages/cli/bin/netscript-dev.ts db <op> [flags]
```

Once the CLI is published on JSR, the same behaviour is available as:

```bash
netscript db <op> [flags]
```

No code changes needed — the transition is seamless.

`packages/cli/bin/netscript-dev.ts` is the local contributor CLI. It exposes maintainer commands
such as `init`, `sync`, `probe`, and `test`, plus project capability commands such as `db`,
`service`, `plugin`, and `generate` against the local checkout. `packages/cli/bin/netscript.ts`
remains the published public-surface equivalent.

### Commands

| Command                    | What it does                                                                                                                        |
| -------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| `db init --name <name>`    | Creates the initial migration and applies it; first-run bootstrap                                                                   |
| `db migrate --name <name>` | Creates a new migration from schema diff and applies it                                                                             |
| `db generate`              | Runs `prisma generate --generator client` + Zod generation + `fix-zod-imports` + `patch-prisma-client`; writes `schema/.generated/` |
| `db seed`                  | Runs `scripts/seed.ts` against the live database                                                                                    |
| `db status`                | Prints Prisma migration status                                                                                                      |
| `db studio`                | Opens Prisma Studio (interactive; keeps Aspire running)                                                                             |
| `db add <engine>`          | Adds a second DB engine to an existing project; regenerates helpers                                                                 |

All commands accept `--project-root <path>` and `--db <config-key>`. When `--db` is omitted and
there is one database, it is used automatically. `--db all` targets all configured databases.

### Required sequence for a fresh project

```
db init → db generate → db seed
```

`db init` must run before `db generate` (the schema must be applied before the client can be
generated). `db generate` must run before any code that imports from `database/<engine>/mod.ts` (the
generated client must exist).

### Schema change workflow

```
(edit schema.prisma) → db migrate --name <description> → db generate
```

## `netscript plugin` — Command Family

`plugin add` composes plugin scaffolding with the DB capability. It never creates a plugin-local
database workspace. DB-backed plugins keep their source schema at
`plugins/<name>/database/*.prisma`; the CLI copies those schema contributions into the selected root
database workspace under:

```text
database/<engine>/schema/plugins/<name>/*.prisma
```

In local contributor mode, run commands directly from a scaffolded project:

```bash
deno run -A packages/cli/bin/netscript-dev.ts plugin add <kind> [flags]
deno run -A packages/cli/bin/netscript-dev.ts plugin list
deno run -A packages/cli/bin/netscript-dev.ts plugin info <name>
deno run -A packages/cli/bin/netscript-dev.ts plugin update <name>
deno run -A packages/cli/bin/netscript-dev.ts plugin remove <name>
deno run -A packages/cli/bin/netscript-dev.ts plugin doctor
deno run -A packages/cli/bin/netscript-dev.ts generate plugins
deno run -A packages/cli/bin/netscript-dev.ts marketplace search workers
deno run -A packages/cli/bin/netscript-dev.ts marketplace publish plugins/workers
```

### `plugin add`

Supported plugin kinds:

| Kind      | Starter name        | Official local-copy name | Notes                                                                                         |
| --------- | ------------------- | ------------------------ | --------------------------------------------------------------------------------------------- |
| `worker`  | any kebab-case name | `workers`                | Copies `plugins/workers`, `workers`, `packages/plugin-workers-core`, and stream dependencies  |
| `saga`    | any kebab-case name | `sagas`                  | Copies `plugins/sagas`, `sagas`, `packages/plugin-sagas-core`, and references `workers-api`   |
| `trigger` | any kebab-case name | `triggers`               | Copies `plugins/triggers`, `triggers`, `packages/plugin-triggers-core`, and worker references |
| `stream`  | any kebab-case name | `streams`                | Copies `plugins/streams` and `packages/plugin-streams-core`                                   |

Flags:

| Flag                    | Purpose                                                                                          |
| ----------------------- | ------------------------------------------------------------------------------------------------ |
| `--name <name>`         | Plugin name; required for deterministic non-interactive use                                      |
| `--port <port>`         | Plugin API port override                                                                         |
| `--service-refs <refs>` | Comma-separated service references                                                               |
| `--plugin-refs <refs>`  | Comma-separated plugin references                                                                |
| `--db <engine-or-key>`  | Target an existing DB key/name or provision a DB engine                                          |
| `--no-db`               | Skip database provisioning and DB wiring                                                         |
| `--samples`             | Scaffold/copy sample tasks, jobs, sagas, and triggers; default                                   |
| `--no-samples`          | Skip sample task/job/saga/trigger material                                                       |
| `--source <mode>`       | `auto`, `starter`, or `local`; `local` copies first-party plugin source from a monorepo checkout |
| `--project-root <path>` | Project root directory                                                                           |
| `--force`               | Overwrite generated files if they already exist                                                  |

Official workers scaffold details:

- Worker config helpers come from `@netscript/plugin-workers-core/config`, not `@netscript/config`.
- Job definitions and handlers come from `@netscript/plugin-workers-core`.
- Runtime registries are generated by the workers plugin CLI from plugin-owned
  `scaffold.runtime.json` manifests. Do not call or recreate a root `scripts/generate-registries.ts`
  path.
- After adding plugins, run `generate plugins` to refresh `.netscript/generated/`.
- `plugin list`, `plugin info`, and `plugin doctor` read discovery output; `plugin update` and
  `plugin remove` dispatch through the same framework verb surface.

### DB selection rules

1. `--no-db` wins and disables DB wiring.
2. `--db <existing-key-or-name>` reuses that configured database.
3. `--db <engine>` provisions that engine when it is not already configured.
4. If the provider requires DB and the project has a primary DB, it reuses the primary DB.
5. If the provider requires DB and no DB exists, it provisions Postgres by default.

After adding a DB-backed plugin, run the DB workflow before booting resources:

```bash
deno run -A packages/cli/bin/netscript-dev.ts db init --db <key> --name init
deno run -A packages/cli/bin/netscript-dev.ts db generate --db <key>
deno run -A packages/cli/bin/netscript-dev.ts db seed --db <key>
```

## DB CLI Mode — How Delegation Works

This is the core mechanism behind every `netscript db` command.

```
deno run -A packages/cli/bin/netscript-dev.ts db <op> --db <key>
        │
        ▼
DbOperationRunner
  sets env vars:
    NETSCRIPT_PRISMA_OPERATION = <op>       (init / generate / seed / status / …)
    NETSCRIPT_PRISMA_TARGET    = <key>      (postgres / mysql / all)
    NETSCRIPT_PRISMA_NAME      = <name>     (init / migrate only)
        │
        ▼
aspire start --apphost <root>/aspire/apphost.ts --isolated
  (env vars are inherited transitively into the TS AppHost subprocess)
        │
        ▼
db-cli-mode.ts (inside the TS AppHost, at startup)
  reads process.env.NETSCRIPT_PRISMA_OPERATION
  if set → registers Executable resource:
    name:    prisma-<op>-<config-key>
    command: deno task db:<op>:<engine>     (runs inside database/<engine>/)
    env:     DATABASE_URL + <ENGINE>_URI   (from Aspire DB resource reference)
    waitFor: the DB container resource
        │
        ▼
CLI polls: aspire ps --resources
  waits for prisma-<op>-<config-key> → state: Finished | Failed
  then fetches: aspire logs prisma-<op>-<config-key>
  prints resource logs to console
        │
        ▼
aspire stop (always, in finally block)
```

Env vars are the sole trigger mechanism. The AppHost reads them via `process.env` at startup — no
CLI arg forwarding or Aspire config bridging.

## The `.helpers/` Layer

`.helpers/` is fully generated by the CLI. It is the TypeScript implementation of what
`dotnet/.generated/*.g.cs` is in the framework monorepo — the per-project Aspire wiring layer.

| Helper file                  | What it generates                                                                                     |
| ---------------------------- | ----------------------------------------------------------------------------------------------------- |
| `configure-dashboard.ts`     | Dashboard OTLP environment setup                                                                      |
| `register-infrastructure.ts` | `builder.addPostgres/addMySql/addSqlServer` calls; skips sqlite (file-backed)                         |
| `register-services.ts`       | One `builder.addExecutable` per `NetScript.Services` entry, wired to the primary DB and to each other |
| `register-plugins.ts`        | Plugin API resources, DB/KV/service/plugin references, and two-pass plugin reference wiring           |
| `register-background.ts`     | Worker/saga/trigger background processors                                                             |
| `register-apps.ts`           | `builder.addExecutable` for the dashboard (and any other apps)                                        |
| `register-tools.ts`          | `builder.addExecutable` for Prisma Studio and other dev tools                                         |
| `db-cli-mode.ts`             | Short-circuit: if `NETSCRIPT_PRISMA_OPERATION` is set, register the CLI executable and return early   |

Regenerate `.helpers/` with the owning capability command after any topology change:

```bash
deno run -A packages/cli/bin/netscript-dev.ts service generate --project-root <root>
deno run -A packages/cli/bin/netscript-dev.ts plugin add <kind> --name <name> --project-root <root>
deno run -A packages/cli/bin/netscript-dev.ts db add <engine> --project-root <root>
```

Do not edit helper files directly; they are CLI output.

## Database Facade — `database/<engine>/mod.ts`

The generated `mod.ts` exposes a unified `db` object:

```typescript
import { db } from './database/postgres/mod.ts';

db.postgres; // typed Prisma client
db.client; // same, engine-agnostic alias
db.getClient(); // Promise<PrismaClient>
db.disconnect();
db.healthCheck();
```

The facade is the import target for:

- Service `main.ts` — passed as `db` option to `defineService()`
- Service handlers — received as `context.db`
- Seed scripts
- Any code that needs direct DB access

Connection string is resolved at first use by reading `<ENGINE>_URI` then `DATABASE_URL` from the
environment. Inside Aspire, those env vars are injected automatically via `withEnvironment` and
`withReference`. Outside Aspire (direct `deno run`), export `DATABASE_URL` manually.

The import alias `@database` can be registered in a service's `deno.json`:

```json
"imports": {
  "@database": "../../database/postgres/mod.ts"
}
```

## Wiring Pattern: Schema → Service → App

The scaffold creates the skeleton. You extend it in three layers.

### Layer 1 — Schema (database)

Add Prisma models to `database/<engine>/schema/schema.prisma`. Run `db migrate` + `db generate`
after every schema change. The generated `schema/.generated/zod/` provides type-safe Zod schemas for
every model — import them in contracts and handlers.

### Layer 2 — Service (backend)

Services are Hono + oRPC processes. Each one:

1. Has its own `deno.json` workspace member under `services/<name>/`
2. Imports `@database` for the Prisma client
3. Imports `@contracts` for the oRPC contract it implements
4. Calls `defineService(router, { db: db.<engine>, port, name })` in `main.ts`
5. Is registered in `appsettings.json` under `NetScript.Services`
6. Is added to the `workspace` array in the root `deno.json`

Aspire then picks it up: `register-services.ts` creates an Executable resource that inherits
`DATABASE_URL` from the DB reference and is waited on by any downstream resource that declares a
dependency.

Service URLs are injected by Aspire as:

```
services__<name>__http__0 = http://localhost:<port>
```

Cross-service calls use `createServiceClient` from `@netscript/sdk` which reads this env var
automatically.

### Layer 3 — App (frontend)

The dashboard is a Fresh 2 app. To expose a service's data in the UI:

1. Create an oRPC client with `createServiceClient` in `apps/dashboard/lib/`
2. Add a route under `apps/dashboard/routes/(dashboard)/`
3. Use the `definePage() → .layer() → .build()` builder for server-rendered pages
4. Use the `DeferPage` + partial pattern for streamed data
5. Add nav entries in the dashboard layout

The Fresh app receives `services__<name>__http__0` from Aspire automatically — no manual URL
configuration.

## Contracts

Contracts are the seam between services and the frontend. Always define the contract before writing
the handler or the client.

```
contracts/versions/v1/<entity>.contract.ts
    ↓ export from contracts/versions/v1/index.ts
    ↓ included in contracts/versions/v1/mod.ts (v1 object)
    ↓
    ├── Service handler: v1.<entity>.$context<{db}>().<method>.handler(...)
    └── App client:      createServiceClient<typeof contract>({ serviceName })
```

Use generated Zod schemas (`schema/.generated/zod/`) as a base for contract output schemas when the
shape matches exactly. Compose or extend them for API-specific shapes (e.g. pagination wrappers,
computed fields).

## Smoke Test Workflows

### DB-only smoke

After scaffolding a DB-backed project, run this sequence to confirm the DB stack works before
writing application code.

```bash
cd <project>

# 1. One-time Aspire SDK setup
cd aspire && aspire restore && cd ..

# 2. Bootstrap
deno run -A packages/cli/bin/netscript-dev.ts db init --name init
deno run -A packages/cli/bin/netscript-dev.ts db generate
deno run -A packages/cli/bin/netscript-dev.ts db seed
deno run -A packages/cli/bin/netscript-dev.ts db status

# 3. Verify artifacts
ls database/<engine>/schema/.generated/client.server.ts
ls database/<engine>/schema/.generated/zod/schemas/
```

Each command prints full console output (progress, Prisma logs, success/failure status).

### Full scaffold E2E smoke

Before merging changes that affect scaffold output, plugin scaffolding, DB wiring, Aspire helper
generation, or official plugin copy mode, run the full CLI e2e suite:

```bash
deno task e2e:cli run scaffold.plugins --cleanup --format pretty
```

This is the strongest scaffold gate in the repo. It creates a fresh project, initializes it in local
contributor mode, adds the four official first-party plugins (`workers`, `sagas`, `triggers`, and
`streams`), provisions Postgres through `plugin add`, restores Aspire modules, runs `db init`,
`db generate`, and `db seed`, starts Aspire isolated, waits for core resources, calls real plugin
endpoints, triggers a worker job/webhook path, validates OTEL traces, and stops Aspire when
`--cleanup` is set.

Manual runs should use `--format pretty`: the tool prints every step start, pass/fail result, and
long-command heartbeat while the run is active. It also writes an essential progress log to
`.llm/tmp/scaffold-e2e-test/<run-name>.log`. Agent runs may keep the default `ndjson` format for
machine-readable start, progress, step, and summary events.

Omit `--cleanup` only when you intentionally want the generated Aspire runtime left running for
manual dashboard/log/resource/trace inspection. Stop it when finished.

The script intentionally covers the cases unit tests cannot prove:

- local package and official plugin copy workflow
- plugin DB detection/provisioning/reuse
- plugin schema copy into the root DB schema tree
- migration, generated Prisma client, generated Zod schemas, and seed order
- Aspire resource registration for services, plugin APIs, background processors, database, cache,
  and tools
- worker job execution through the generated runtime
- real HTTP endpoints for workers, sagas, and triggers

### Manual full-test scaffold

For local maintainer review, create a full generated project under the gitignored `scaffold/`
directory from the repo root. Use the worktree CLI:

```bash
deno run -A packages/cli/bin/netscript-dev.ts init full-test \
  --path scaffold \
  --db postgres \
  --service --service-name users --service-port 3001 \
  --ci --yes --no-git --force

cd scaffold/full-test
deno run -A ../../packages/cli/bin/netscript-dev.ts plugin add worker --name workers --project-root . --force
deno run -A ../../packages/cli/bin/netscript-dev.ts plugin add saga --name sagas --project-root . --force
deno run -A ../../packages/cli/bin/netscript-dev.ts plugin add trigger --name triggers --project-root . --force
deno run -A ../../packages/cli/bin/netscript-dev.ts plugin add stream --name streams --project-root . --force

deno run -A ../../packages/cli/bin/netscript-dev.ts plugin list
deno run -A ../../packages/cli/bin/netscript-dev.ts plugin doctor --project-root .
deno run -A ../../packages/cli/bin/netscript-dev.ts generate plugins --project-root .
deno task check
```

Expected plugin inventory: `workers`, `sagas`, `triggers`, and `streams`. Use the scaffold-local
`deno task check` as the authored-project validation gate. Raw `deno check --unstable-kv .` may
descend into generated caches such as Aspire's restored `.modules` SDK; if it is run and fails
there, record that generated-cache blocker separately from plugin installation health.

If the script starts Aspire and a later step fails, stop the generated AppHost manually before
retrying:

```bash
aspire ps
aspire stop --apphost <path-to-generated-project>\aspire\apphost.ts --non-interactive --nologo
```

**Aspire inspection**: While the CLI handles the full lifecycle for DB operations, Aspire remains
the best way to inspect a running full stack. Use `aspire describe` for resource state,
`aspire logs <resource>` for console output, and `aspire otel logs` / `aspire otel traces` when the
dashboard API is available. If an Aspire MCP server is enabled in the session, use its equivalent
resource, log, and trace tools for the same checks.

**Important**: Do not skip ahead if an operation fails — the chain is ordered and each step depends
on the previous one.

## Starting the Full Stack

Once the DB is bootstrapped and services are registered:

```bash
cd aspire && aspire start
```

Aspire starts all resources in dependency order:

- DB container (postgres / mysql / …)
- Services (wait for DB healthy)
- Plugin APIs (wait for DB/cache/services/plugins as configured)
- Background processors (workers / sagas / triggers)
- Apps (wait for services)
- Tools (prisma-studio waits for DB)

Monitor with:

```bash
aspire ps --resources --format Json   # resource list + health
aspire logs <resource-name>           # console output
aspire otel logs <resource-name>      # structured logs
```

The Aspire dashboard URL is printed on startup (`https://localhost:18888/…`). It shows distributed
traces, metrics, and structured logs across all resources.

## Tips for LLMs

- **Read `appsettings.json` first.** It tells you which DBs, services, and apps exist. Never assume
  the project has services or plugins — check `"Services"`, `"Plugins"`, and
  `"BackgroundProcessors"`.
- **Use the full scaffold E2E smoke before declaring scaffold work complete.** For any CLI scaffold,
  DB, plugin, or Aspire helper change, run
  `deno run --allow-read --allow-write --allow-run --allow-net --allow-env .llm/tools/scaffold-e2e-test.ts --format pretty`
  or document precisely why it could not run.
- **Run DB commands via the CLI.** Use `deno run -A packages/cli/bin/netscript-dev.ts db <op>`. The
  CLI starts Aspire in isolated mode, polls for completion, fetches and prints resource logs, then
  stops Aspire. Full console output is shown — no need to inspect logs manually.
- **The DB operation chain is ordered.** `init` before `generate`, `generate` before `seed`. If
  `generate` hasn't run, `schema/.generated/client.server.ts` does not exist and any import from
  `mod.ts` will fail with Module not found.
- **`client.server.ts` is framework-generated, not Prisma output.** It is created by
  `runPatchPrismaClient()` called from `fix-zod-imports.ts` during `db generate`. All engines
  produce it uniformly.
- **Always run the schema change sequence atomically.** When you edit `schema.prisma`, run
  `db migrate` and `db generate` in the same session before writing any handler code that references
  the new model.
- **Never edit `.helpers/` directly.** Changes are overwritten when the owning CLI capability
  regenerates helpers (`service generate`, `service add`, `plugin add`, `db add`, and init-time
  generation).
- **Add workspace members before type-checking.** When you create `services/<name>/`, add
  `./services/<name>` to the `workspace` array in the root `deno.json` before running
  `deno task check`.
- **Connection strings are injected by Aspire.** Do not hardcode ports or passwords in service code.
  If you need to run a service directly outside Aspire for debugging, export `DATABASE_URL` manually
  from `docker inspect`.

## Tips for Developers

- Run `aspire restore` once after `netscript init` before any `aspire start`. Without it,
  `.modules/aspire.ts` does not exist and the AppHost crashes.
- `netscript db generate` must be re-run after every `schema.prisma` edit. The generated client is
  in `.gitignore` — teammates must run it locally.
- Prisma Studio is available in the Aspire dashboard as the `prisma-studio` resource. It starts in
  `Waiting` state until the DB container is healthy, then transitions to `Running`.
- When adding a second DB engine with `netscript db add <engine>`, the helpers are regenerated
  automatically. Restart Aspire to pick up the new resource.
- `DATABASE_URL` and `<ENGINE>_URI` are both set in every DB CLI operation and in every service
  process by Aspire. The `prisma.config.ts` in each DB workspace normalises both .NET connection
  string format and postgres:// URL format, so either can be passed.
- The scaffold uses Prisma v7 with driver adapters. `client.server.ts` is created by
  `runPatchPrismaClient()` from `@netscript/database/scripts` — it is NOT a Prisma output but a
  framework post-processing step invoked by `fix-zod-imports.ts`. All engines produce
  `client.server.ts` uniformly. Always import from `client.server.ts`, not `client.ts`.

## Key Files

| File                                         | Role                                                                    |
| -------------------------------------------- | ----------------------------------------------------------------------- |
| `appsettings.json`                           | Single config source for Aspire; drives helper generation               |
| `aspire/apphost.ts`                          | AppHost entry; calls `createNetScriptAppHost()`                         |
| `aspire/.helpers/db-cli-mode.ts`             | DB CLI short-circuit; reads `NETSCRIPT_PRISMA_OPERATION`                |
| `aspire/.helpers/register-infrastructure.ts` | DB + cache registration                                                 |
| `aspire/.helpers/register-services.ts`       | Service executable registration                                         |
| `aspire/.helpers/register-plugins.ts`        | Plugin API executable registration and cross-plugin wiring              |
| `aspire/.helpers/register-background.ts`     | Worker/saga/trigger background processor registration                   |
| `database/<engine>/mod.ts`                   | DB facade (`db.client`, `db.getClient()`, etc.)                         |
| `database/<engine>/schema/schema.prisma`     | Authoritative schema; edit this                                         |
| `database/<engine>/schema/plugins/<plugin>/` | Copied plugin schema contributions                                      |
| `database/<engine>/migrations/`              | Migration history; commit to git                                        |
| `database/<engine>/schema/.generated/`       | Prisma output; do NOT commit                                            |
| `database/<engine>/scripts/seed.ts`          | Replace with real seed data                                             |
| `contracts/versions/v1/`                     | Contract definitions; one file per domain entity                        |
| `services/<name>/src/main.ts`                | Service entry; `defineService(router, { db })`                          |
| `plugins/<name>/database/`                   | Plugin-local source schema contribution                                 |
| `.llm/tools/scaffold-e2e-test.ts`            | Full generated-project smoke gate for scaffold/plugin/DB/Aspire changes |
