## Per-feature evidence + draft pitch (confidence: STRONG unless noted)

1. **oRPC service contracts (`@netscript/service`, contract-first).** Evidence:
   `contracts/versions/v1/channel.contract.ts`, `contracts/versions/v1/skills.contract.ts`,
   `services/eischat/src/routers/channel.ts` (`implement(ChannelContractV1)`),
   `apps/dashboard/lib/channel-service.ts` (typed client built straight off the contract type).
   Pitch angle: one contract drives server handler, typed browser client, and DB shape
   (Zod-from-Prisma) at once — a new contract route is automatically typed on the client, no
   separate "keep the API docs in sync" step.

2. **Background workers / single-writer job architecture (`@netscript/plugin-workers-core`).**
   Evidence: `workers/jobs/embed-document.ts`, `workers/jobs/transcribe-image.ts` — both
   compute-only by design because tursodb holds an exclusive OS file lock per channel db (a second
   process opening it crashes with "os error 33"); results are written back via
   `channelClient.writeKnowledgeChunks`/`readKnowledgeImage`. Pitch angle: the job-handler contract
   made "workers never touch the database directly" the natural default, not something the team had
   to invent.

3. **Durable streams (`@netscript/plugin-streams-core`).** STRONG for the real chat-transport usage
   (`toDurableChatSessionResponse` wrapped in a `durable.write` OTel `PRODUCER` span in
   `apps/dashboard/routes/api/chat.ts`) — a chat turn survives a reload/dropped connection because
   the stream write is decoupled from the POST that started it. **WEAK/NONE for the scaffolded
   `plugins/streams` primitive itself** — `streams/notifications-stream.ts` is confirmed to be the
   shipped stub only, unused product code. Do not present the stub as a proof point; use the
   chat-transport usage instead.

4. **Channel sync / plugin swap seam.** Evidence: `docs/PHASE-6-SHARING.md`,
   `services/eischat/src/sharing.ts` — `ChannelSync` interface (`list`/`pull`/`publish`), one
   implementation shipped (`SmbRegistrySync`, an SMB file share) with a named future implementation
   already slotted in (`TursoSync`). Pitch angle: shipped the version that works today without new
   infrastructure, while the same seam leaves room to swap in a different sync backend later without
   touching the caller.

5. **Prisma-on-Deno database layer.** Evidence: `database/sqlite/mod.ts` — `PrismaLibSql` driver
   adapter (Prisma 7 requires one; Deno-native, no node addon), lazy singleton,
   `enablePrismaTracing()` gated on `OTEL_DENO`. Pitch angle: real Prisma running natively in Deno
   with no node-gyp/native-addon shimming; reused NetScript's DB-layer conventions instead of a
   bespoke Prisma bootstrap for this one app.

6. **Aspire orchestration + generated `.mts` helpers.** Evidence: `aspire/.helpers/
   register-services.mts` (128 lines), `register-background.mts` (165 lines) — both headed
   "GENERATED BY @netscript/cli. DO NOT EDIT," both containing extensive deliberately-preserved
   hand-edited sections. Pitch angle: the entire local dev topology (7 services/processors, health
   checks, ports, cross-service discovery) generated from declarative config and regenerated on
   every scaffold change — real evidence the generator works end-to-end, but also the honest record
   of where its coverage stops short (see gaps below).

7. **Generative UI / "NS One" skill system.** Evidence: `services/eischat/src/builtin-skills.ts`
   (459 lines) — the `native-ui-composition` builtin skill: full data-shape-to-component mapping
   table (stat/donut/area/chart/table/timeline/flow/heatmap/tree/kanban/list/keyvalue/avatar/callout),
   layout budget rules, a worked example literally titled "VIF estate health," plus a sandboxed
   `\`\`\`widget` HTML escape hatch. Pitch angle: the model can emit a fenced JSON tree that renders
   using the app's own design tokens, taught via a shipped onboarding skill whose worked example is
   the product's own real domain — not bolted on afterward.

8. **MCP grounding (legacy-archeo-mcp).** Evidence: `docs/PHASE-3-MCP.md` — curated tools
   (`explain_entity`, `lineage`, `findings`, `dotnet_deployments`, `scheduler_activity`,
   `vif_dependencies` — explicitly commented "VIF→CSB work"), grounded text-to-SQL restricted to a
   documented `v_*` view layer ("~90 views... the semantic layer"); explicit boundary — "No direct
   DuckDB access from eis-chat... the boundary is the MCP server." Pitch angle: the model gets real
   query power over a decades-old legacy estate without raw table access.

9. **Skills system (progressive disclosure).** Evidence: `contracts/versions/v1/skills.contract.ts`
   (168 lines), `services/eischat/src/skills-git.ts` (330 lines, pure fetch/regex GitHub `SKILL.md`
   importer, no yaml dependency, github.com-host-restricted), three seeded builtin skills. Pitch
   angle: a user can import any public GitHub repo's `SKILL.md`; the chat route resolves only the
   right skill bodies per turn (tag or semantic match) instead of stuffing every skill into every
   prompt.

10. **Telemetry / chat-turn tracing.** Evidence: `apps/dashboard/routes/api/chat.ts` (637 lines) —
    real span tree `chat.turn` (root, continues browser trace) → `kb.search` (per tool call) →
    `durable.write` (PRODUCER), real GenAI semantic-convention attributes (`gen_ai.request.model`,
    `gen_ai.request.effort`, `gen_ai.usage.output_tokens`). Pitch angle: every chat turn is traced
    end-to-end from browser through model call to the durable-stream write — the same trace tree a
    support engineer opens to see exactly what a wrong answer did, tool call by tool call. **Caveat:
    SigNoz cross-server correlation (`chart_incidents_by_dotnet_program`) is explicitly deferred**
    per `docs/PHASE-7-MCP-UI.md` — report as roadmap, not shipped.

11. **Desktop packaging (`deno desktop`).** Evidence: `docs/PHASE-1-FOUNDATION.md` — one Fresh entry
    point (`main.ts`) serves `deno task dev`, `deno desktop --hmr` (native window), and
    `deno desktop -o dist/` (release binary) identically; native-only chrome feature-detected via
    `"BrowserWindow" in Deno`. **Doc-sourced, not code-verified this pass** — treat as strong-but-
    unconfirmed against the actual shipped `main.ts`.

12. **Chat provider seam / architecture-drift transparency.** Evidence: `docs/PHASE-2-CHAT.md`
    (original plan: single LiteLLM gateway with BYOK virtual keys) vs. `apps/dashboard/lib/llm.ts`
    (217 lines, actual shipped code: LiteLLM gateway "hasn't been wired"; four dedicated TanStack AI
    adapters instead — Anthropic, OpenAI, OpenRouter [chosen specifically to route around a real
    upstream regression, TanStack/ai#342, verified via a localhost capture server], Ollama [local/
    keyless, `GET /api/tags` preflight returning a clean 503 instead of a mid-stream crash]). Pitch
    angle: this is a plainly-stated architecture-drift note in the code's own top comment — exactly
    the factual-callout voice the docs should use, no "honesty" framing needed because the code
    itself already models it.

13. **Contract-driven codegen bridging Prisma+Zod.** Same evidence as item 5; the generic
    postgres/mysql/mssql connection-string normalizers live in the same facade module even though
    eis-chat only uses sqlite — flag this as framework-level reuse evidence, not eis-chat-specific
    usage, if cited.

## The two narrative spines (richest material — use as story templates)

### VIF → CSB ERP migration (grounding/MCP story)

One sentence framing recurs across the docs: eis-chat exists to prove NetScript's core loop "for one
real team facing one real problem (the VIF → CSB migration)" (`docs/PRODUCT.md`). VIF is a legacy
internal ERP/production system; CSB is its planned replacement. Named target users: Eric, Benoit,
Arnaud. The MCP tool `vif_dependencies(program?)` exists *because of* this migration (commented "VIF→
CSB work" in `docs/PHASE-3-MCP.md`), the generative-UI skill's worked example is titled "VIF estate
health," and the RAG pipeline (`docs/PHASE-4-RAG.md`) exists specifically so CSB migration docs can
be dropped into a channel and queried later ("what makes Arnaud's CSB-docs use case work"). This is
not a marketing hook bolted on after the fact — it's the literal design brief that shaped which MCP
tools, RAG use case, and UI example got built.

### PROSCO / Prolabel incident diagnosis (composability story — the richest single example)

Documented causal chain in `docs/PHASE-5-CONTEXT.md`:
`Benoit pastes a PROSCO alert-email screenshot → vision model reads it, extracts {program, IMP
number, table, timestamp, error text} → agent calls legacy-archeo-mcp: explain_entity/lineage →
PROSCO → Prolabel flow → (when available) signoz-mcp: matching error trace around the timestamp →
agent returns a timestamped, evidence-linked diagnosis.`

Every step is backed by real code: the vision-extraction step is `workers/jobs/transcribe-image.ts`
(single-writer-compliant — never opens the channel DB directly, fetches bytes back via
`channelClient.readKnowledgeImage`); this is also where a real documented NetScript-adjacent bug was
fixed (`#kb-image-payload` — raw image bytes in the job payload blew Deno KV's 64KB enqueue cap;
fixed by carrying only a small reference through the queue and fetching bytes back via RPC). The MCP
step uses the same curated tool surface as the VIF story. The SigNoz step is **explicitly deferred**
— report as roadmap only. The final diagnosis-return step is backed by the same OTel-traced,
tool-calling chat loop as item 10 above. A manual-promotion step (`promoteToChannel(...)`) turns an
ephemeral diagnosis into durable, searchable per-channel knowledge for next time.

**This is the single richest evidence of "turns saved" by NetScript-adjacent primitives composing
together** — a vision-parsing worker job, a single-writer DB architecture, a curated MCP boundary, an
OTel-traced chat loop, and a manual-promotion memory path, all composing into one paste-a-screenshot-
get-a-diagnosis interaction. Use this as the template "story spine" the per-feature authoring briefs
should emulate (parallel to `tutorials/storefront/04-checkout-saga.md`).

## Confirmed gaps (do not use as positive proof points; report as open items if referenced at all)

- **Auth/authorization: total confirmed absence** in eis-chat's own code — only vendored third-party
  docs or NetScript's own reference docs (describing the capability, not eis-chat's usage of it)
  matched a repo-wide grep. Do not imply eis-chat proves NetScript's auth story.
- **`compileWorkersRegistry` emits only a handler map, not a job-definition map** — forces a
  hand-maintained project-owned shim (`workers/job-definitions.ts`); confirmed still present at
  0.0.1-beta.1 per the file's own comment. Real, current, plainly-stated friction.
- **workers-API service exposes only OpenAPI routes, not RPC** — breaks the typed `@netscript/sdk`
  client for job-triggering; a landmine where job `id` must be in the request body or a created
  task's id becomes `undefined`.
- **A `Service` cannot declare `PluginReferences`** (only Apps/BackgroundProcessors can) — forces
  manual hand-edited env-wiring in generated Aspire helpers that must survive every regen; concrete
  fallout includes a generator perm-flag regression, a split-brain-KV bug (rickylabs/netscript#371),
  and a documented event-drop bug in the workers plugin's `DurableStreamProducer`.
- **SigNoz cross-server correlation is explicitly deferred**, not shipped.
- `SKILL.md`/`HANDOVER.md` describe an earlier, now-stale repo layout — don't cite them for current
  structure.
