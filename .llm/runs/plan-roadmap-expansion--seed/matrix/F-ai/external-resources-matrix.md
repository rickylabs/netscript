# Topic F-ai — External Resources Matrix (Stage-B)

Master matrix of the external AI ecosystem the F-ai suite should wrap. Legend — `category`:
`sdk-streaming` (provider abstraction / streamText / useChat) · `tool-protocol` (MCP) ·
`adapter-pattern` (how N-provider abstraction is architected) · `telemetry-semconv` (GenAI OTel).

| resource | category | Deno/JSR fit | license | maturity | what-to-wrap | risk |
|---|---|---|---|---|---|---|
| **TanStack AI** (`@tanstack/ai`, `@tanstack/ai-anthropic`, `@tanstack/ai-openai`, `@tanstack/ai-mcp`) | sdk-streaming | **Proven today** — plain npm packages, no Node-only API observed, already imported via `npm:` specifiers in `packages/ai/deno.json` and passing `deno check`/publish-dry-run | MIT | Pre-1.0 (`^0.39.0` core, `0.2.1` exact-pinned for `ai-mcp`); 1,300+ releases, active (last release seen same-day); self-described alpha by third-party reviews despite release volume | **Already wrapped** by `packages/ai/src/adapters/{anthropic,openai-compatible}.adapter.ts` behind NetScript's own `ChatClientPort`/`ChatModelProviderPort` seams — no TanStack type crosses the public boundary (`toTanstackChatClient` anti-corruption). Reaffirm; do not swap for Vercel AI SDK. | Pre-1.0 churn: breaking upstream releases can land inside a caret range (`^0.39.0`); the exact-pin on `ai-mcp@0.2.1` shows the pattern is already known and defended against for at least one subpath — extend exact-pinning discipline to the others if churn is observed. |
| **Vercel AI SDK** (`ai`, `@ai-sdk/openai`, `@ai-sdk/anthropic`, `@ai-sdk/google`, `useChat`) | sdk-streaming | **Weak** — docs require Node.js 22+; community reports (Supabase Edge/Deno Deploy threads) describe `npm:`-specifier friction and note Vercel does not target Deno as a first-class runtime | MIT (Apache-2.0 for some sub-packages historically; verify per-package before any adoption) | High — 25.4k GitHub stars, 7,460+ commits, 25+ providers, AI SDK 5/6 already shipped, `stopWhen`-based tool-calling loops, structured-object streaming, reranking, embeddings, image editing | **Do not wrap as the primary engine** — confirms the existing TanStack choice. Worth a **narrow future look** only for two gaps TanStack doesn't cover yet: structured-object streaming and embeddings/reranking primitives, if `@netscript/ai`'s `EmbeddingProviderPort` (currently throwing-default, E6) ever needs a second reference shape. | If a future slice imports any `@ai-sdk/*` package for one primitive, it must NOT cross NetScript's public AI surface (same anti-corruption rule already enforced for TanStack) — otherwise two competing provider-SDK vocabularies leak into `@netscript/ai/contracts`. |
| **MCP TypeScript SDK** (`@modelcontextprotocol/sdk`) | tool-protocol | Good in principle (MIT, pure TS, transports are HTTP/stdio-based); NetScript does **not** import it directly today — `packages/ai/src/ports/mcp-transport.ts` is a hand-written port designed to be backed by `@tanstack/ai-mcp` as the concrete adapter, not the official SDK | MIT (main package); repo moving to Apache-2.0 for new contributions | Approaching v2 — spec revision `2026-07-28` due, v1.x supported ≥6 months post-v2; peer-deps on Zod v4 (back-compat to 3.25) | The **port design** (`McpTransportPort`, connection-state machine, backoff config, `stdio`/`streamable-http` transport kinds) already mirrors the official SDK's shape — good precedent-following even without a direct dependency. If a second/alternate MCP adapter is ever built (bypassing TanStack), wrap this SDK directly rather than hand-rolling the JSON-RPC/transport layer. | The official SDK's spec-version churn (v1→v2 mid-2026) means any future direct adapter must pin a spec version explicitly; `@tanstack/ai-mcp` insulates NetScript from this today by being the sole concrete implementation. |
| **`@tanstack/ai-mcp`** (TanStack's MCP client/tool-bridge, wraps the protocol for TanStack AI's tool-calling loop) | tool-protocol | Same npm-via-Deno fit as core TanStack AI; exact-pinned (`0.2.1`, not caret) in `packages/ai/deno.json` — the strictest pin in the package, signalling known fragility | MIT | Early (0.2.x) | **Already the concrete backer** referenced in `mcp-transport.ts`'s module doc ("Concrete adapters may wrap `@tanstack/ai-mcp`"). Confirm this is implemented (not just documented) before the F-ai plan assumes MCP support is live — `--mcp` scaffolding is explicitly deferred (`plugins/ai` README, tracked in issue #290). | Two-layer indirection (NetScript port → TanStack MCP → official protocol) adds a translation surface; verify `McpToolDescriptor`/`McpToolResult` field-for-field parity with the official SDK's `Tool`/`CallToolResult` shapes so nothing is silently dropped. |
| **Provider-adapter pattern — library-composition** (TanStack AI's own shape; NetScript's `ChatModelProviderPort` + self-registering adapters) | adapter-pattern | N/A (pattern, not a dependency) | N/A | Proven — this is the exact shape `@netscript/ai` already implements (`registerModelProvider` side-effect registration, mirrors `@netscript/kv/redis`) | **Already correctly wrapped as a pattern**, not a library. The N-adapter model (one `ModelProviderPort` implementation per provider, zero-dependency core, bundle-isolated subpaths) is the right shape for `plugin-ai-core` + adapters; no change needed. | None distinct from the sdk-streaming risk above — the pattern is sound, only the underlying SDK maturity is the open risk. |
| **Provider-adapter pattern — gateway/proxy** (LiteLLM: single `completion()` call normalizing 100+ providers to OpenAI wire format; conceptually same as an "AI gateway") | adapter-pattern | Poor direct fit — LiteLLM is Python-only; no TS/Deno port exists | MIT | High maturity or the gateway pattern generally | **Do not adopt the library** (wrong language) but the **pattern is already partially present**: `packages/ai/src/adapters/openai-compatible.adapter.ts` is NetScript's own "normalize to OpenAI-compatible wire format" adapter (works against DeepSeek/Together/vLLM/any OpenAI-shaped endpoint). Recognize this as the gateway-pattern analogue already implemented — no separate gateway package needed for v1. | If NetScript ever wants spend-tracking/guardrails/load-balancing (LiteLLM's actual value-add beyond normalization), that is out-of-scope infrastructure, not an SDK concern — flag as a future ops/telemetry topic, not an F-ai library dependency. |
| **OpenTelemetry GenAI semantic conventions** (`gen_ai.*` attributes: `operation.name`, `provider.name`, `request.model`, `usage.input_tokens`/`output_tokens`, `response.finish_reasons`; span name `{operation.name} {request.model}`) | telemetry-semconv | Deno-native OTel (`OTEL_DENO=1`) already used elsewhere in the repo (per Topic-B research); GenAI spans specifically arrive **for free** via TanStack AI's own middleware, not a separate NetScript instrumentation layer | CC-BY-4.0 (spec text); code samples MIT-equivalent | **Development** status — explicitly not yet stable; spec moved to its own `semantic-conventions-genai` repo; `OTEL_SEMCONV_STABILITY_OPT_IN` env var gates old-vs-new attribute shapes during the transition | **Wrap, don't reinstrument.** `@tanstack/ai/middlewares/otel` (`chatOtelMiddleware`) already emits `gen_ai`-semconv-shaped spans in eis-chat's real code today (`chat <model>` root span, per-iteration `chat <model> #N`, `execute_tool <name>` per tool call, real `gen_ai.usage.*`/`finish_reasons`). F-ai's telemetry story should document/require wiring this middleware rather than hand-writing GenAI spans against `@netscript/telemetry` directly. Ties directly into Topic-B's OTel semconv research (`matrix/B-telemetry/INDEX.md` §1). | Development-status attributes can rename/reshape before stabilizing (2026 roadmap has no stabilization date) — pin the `OTEL_SEMCONV_STABILITY_OPT_IN` value NetScript targets and re-check on any TanStack AI middleware bump. Also: eis-chat's `tools/legacy-archeo-mcp.ts` (a real MCP server) currently has **zero telemetry** — the GenAI span story stops at the tool-call boundary and does not yet cross into MCP-server-side spans; this is a real gap, not solved by the TanStack middleware. |
