# Prior-art matrix — unified-platform, packaging, updaters, offline-sync

| Prior art | What it is | Maturity | Relevance to topic E | Depth file |
| --- | --- | --- | --- | --- |
| **`@netscript/service` "RFC 14 unified-platform seam"** | In-repo: `createService(router,...).build()` returns a non-listening `ServiceApp` (a `Hono` instance, callable via `.fetch(request)`) instead of only `serve()` (real Deno listener). Docstring explicitly: "keeps the RFC 14 unified-platform seam open for callers that mount service apps into another host." | **Already implemented, shipping today** in `@netscript/service`. | This is the load-bearing server-side primitive for in-process/single-process hosting of multiple NetScript services in one process — exactly what desktop single-process mode (option c) needs. | `analysis/E-desktop/sdk-link-mode-and-service-seam.md` |
| **NetScript issue #349 "RFC-14 unified-mode + Nitro `deno_server`"** | GitHub-tracked epic-child: tier-3 serverless (Vercel/CF/Netlify) via Nitro v3 + oRPC + Fresh 2. | WATCH / deferred (`wave:defer`, milestone Backlog/Triage). | A *different, larger* consumer of the same underlying "RFC 14 unified platform" idea, aimed at serverless bundling rather than desktop. Its WATCH status does **not** block the (already-shipped) service-seam or a scoped sdk link-mode feature. See drift note in `analysis/E-desktop/sdk-link-mode-and-service-seam.md`. | same |
| **Nitro `deno_server` preset** (nitro.build) | Framework preset that runs a Nitro server build inside the Deno runtime. | Requires `--unstable`; documented invocation is Node-emulation-on-Deno (`deno run --unstable --allow-net --allow-read --allow-env .output/server/index.ts`), not native `Deno.serve`. | Confirms the topic spec's characterization: Nitro's Deno story is not yet a clean native-Deno single-process target; it is a separate porting layer, distinct from `deno desktop`. | `research/E-desktop/rfc14-nitro-packaging-prior-art.md` |
| **Tauri v2 updater** | `latest.json` manifest + Ed25519/Minisign-signed artifacts; app polls, downloads, verifies pubkey, installs, restarts. | Mature, production-grade, cross-platform (incl. Windows apply-in-place). | Direct comparison point for `Deno.autoUpdate()`'s Windows gap (stages but does not apply). Tauri's Windows story works because it uses a separate installer/relauncher process, not in-place DLL swap. | `research/E-desktop/rfc14-nitro-packaging-prior-art.md` |
| **Electron autoUpdater / Squirrel** | Similar manifest+signature updater pattern, longer-lived prior art. | Mature. | Same category as Tauri; both demonstrate that Windows in-place update is solved by relaunch/installer indirection, a workaround NetScript could adopt or document as a known Deno limitation. | same |
| **Turso Sync** (successor framing to libSQL "embedded replicas") | Local SQLite (tursodb) file that pushes local mutations to Turso Cloud and pulls remote pages; explicit "Last-Push-Wins" conflict policy with a `transform` hook for custom merges. | Actively developed / GA-track in 2026. | Directly answers "what does offline-first mean for NetScript": a **single local writer per device** (one tursodb file, one process) synced to cloud. This maps cleanly onto single-process desktop (one process = the one local writer) but does **not** solve eis-chat's *within-a-machine* multi-process single-writer contention — that constraint is orthogonal to cloud sync. | `research/E-desktop/turso-sync-offline-first.md` |
| **eis-chat DESKTOP-SHELL.md option ladder (a/b/c)** | In-repo spike doc; (b) adopted for Phase-1 MVP (dashboard-only desktop + separately-run services over 127.0.0.1). | (a) unproven/high-risk (native-addon-in-VFS); (b) shipped; (c) architecture change, out of scope for Phase 1. | The direct precedent NetScript should generalize from — matches `deno desktop`'s documented single-process model exactly at tier (b). | `analysis/E-desktop/eis-chat-desktop-shell-options.md` |
| **NetScript issue #371 (closed) — shared Deno KV Connect** | Restores a shared-KV Aspire resource for **multi-process** mode; clarifies that **local in-process Deno KV is the correct default for unified/single-process deployment**. | Closed/shipped; live-validated via sibling issue #372 (Garnet executable). | Gives NetScript its own working definition of "unified (single-process) vs multi-process" at the KV/queue layer — a smaller, already-solved instance of the same topology question topic E asks for desktop. | `analysis/E-desktop/offline-first-surface.md` |
