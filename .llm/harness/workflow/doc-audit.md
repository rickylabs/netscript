# Doc-Audit Profile — Opposite-Family Single-Pass Audit + Fable Prose Polish for Generated Docs Changesets

Owner-ratified 2026-07-17 (in-session; revised same session). This profile governs how a
documentation changeset produced by Claude generators is audited and polished before merge. It is
referenced from [`lane-policy.md`](./lane-policy.md) (the `docs_audit` and `docs_polish` lane
bindings) and backed in data by the `docs_audit` and `docs_polish` lanes in
`../../tools/agentic/runtime/routing-policy.ts`. The gate set and pipeline live **here**;
`lane-policy.md` points at this file rather than duplicating it.

## Pipeline

The docs changeset moves through a fixed sequence:

1. **Generate** — Claude authoring lanes (single/few **Fable 5 · high** sub-agents, or a Claude
   workflow fleet on **Opus 4.8 · medium** / **Sonnet 5 · high**) produce the changeset.
2. **Audit** — a single **opposite-family** pass by **Codex · GPT-5.6 Sol · medium** over the whole
   changeset, gates executed from evidence, findings and a `## Gate log` recorded.
3. **Fix cycle(s)** — the fixes are made by the **same generator session** (resumed) on the **same
   model** that authored the changeset. Reusing its own context keeps fixes cheap and prevents
   rewrite-drift by an agent that never saw the original authoring decisions. Spawning a **fresh**
   fixer is the exception and MUST be justified in the run log (e.g. the original session was lost).
   Re-audit after fixes; a second audit FAIL triggers the 2-failure escalation below.
4. **Polish** — a single edit-only prose pass by **Claude · Fable 5 · medium** after the audit and
   after fixes have landed.
5. **Merge.**

Stated as a sequence: **generate (Claude authoring lanes) → single-pass Sol audit → fixes by the SAME generator
session (resumed) → repeat audit cycle if needed (2-failure escalation) → single-pass Fable · medium
polish (edit-only) → merge.**

The two lanes are distinct on purpose: the audit is an accuracy gate run by the opposite model
family; the polish is a prose pass by the Claude family. They never merge into one step.

## Trigger

Any docs changeset generated by Claude sub-agents — whether a single or few **Claude · Fable 5 ·
high** authoring sub-agents, or a **Claude workflow fleet** (**Opus 4.8 · medium** / **Sonnet 5 ·
high** authoring lanes) — enters this profile before merge. The trigger is the _changeset_, not the
individual authoring agent.

## Rule: one single audit pass over the entire changeset

The changeset is audited by **Codex · GPT-5.6 Sol · medium** in **ONE SINGLE PASS over the entire
changeset** — never one audit per authoring sub-agent. The auditor MAY use **`high` effort for large
changesets**: this is the declared `large_changeset` effort escalation on the `docs_audit` route in
`routing-policy.ts` (resolved via `resolveCanonicalRouteEffort`), not free-floating discretion —
state the effort chosen and the escalation condition in the run's audit artifact.

**Rationale (the audit unit is the changeset).** The failure modes that actually matter — baseline
drift, cross-page contradictions, and false completeness claims — only exist at _changeset_ scope. A
per-agent audit grades each file "locally fine" and cannot see them: every file can pass in
isolation while the set as a whole is wrong. This is not hypothetical. In the beta.10 docs refresh, a
whole-set audit caught the changeset having been authored against a tree that was missing the release
union — a defect invisible to any single-file or single-agent review, and visible only when the
entire changeset was read against the live baseline in one pass.

## Invariant: opposite-family audit; generator-session ≠ audit-session (hard)

The audit lane is **opposite-family by design**: Claude generators are audited by a **Codex/OpenAI**
session, which restores family diversity for generated-docs accuracy. The audit session is therefore
never a generator session — a hard invariant, identical in force to the harness
generator-≠-evaluator rule, and here satisfied structurally by the family split (there is no
cross-family fallback that could erode it).

**Every accuracy gate is executed by the auditor.** The auditor does not accept the generator's
claims; it runs the commands, `deno doc`s the APIs, and builds site output itself. Verdicts come from
executed evidence, never from the generator's assertions.

## Polish doctrine (the `docs_polish` lane)

After the audit and after fixes land, **Claude · Fable 5 · medium** runs a single final prose-polish
pass. Its doctrine:

- **Edit in place** for prose quality — voice, flow, precision. Fable **must not rewrite documents
  from scratch or burn tokens re-authoring** an already-acceptable document.
- **The one exception** is a document the **audit judged bad enough to warrant a rewrite** — and that
  judgment must appear in the **audit findings**, not be the polisher's own call. Absent an audit
  finding, the polisher edits; it does not re-author.
- **Polish never changes technical claims.** Any accuracy doubt the polisher forms goes **back to the
  `docs_audit` lane**; the polisher does not adjudicate correctness. Its remit is prose, not truth.

**Fallback chain (depth 2), in order.** If the Fable 5 · medium session hits its token ceiling, the
polish falls back to **Claude · Opus 4.8 · xhigh** (Claude-family). Only if **no Claude-agent surface
is available at all** does it fall back — last resort — to **GLM 5.2 · xhigh** over the
`claude-openrouter` transport the design lanes already use. **GLM is a polish-fallback-of-last-resort
only here**; this does not widen GLM beyond its design scope anywhere else in the policy.

## Gate set (audit lane)

Each gate below is executed by the auditor over the changeset (not asserted from generator output).
Each row states what it checks.

| Gate                           | What it checks                                                                                                                                                                                          |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `deno task docs:links`         | Internal documentation links resolve; no broken cross-references introduced or left dangling by the changeset.                                                                                          |
| Site build (Lume) clean        | The documentation site builds cleanly with the changeset applied — no Lume build errors or warnings that fail the build.                                                                                |
| Internal-wording grep          | On **changed lines only**: no issue/PR numbers, no internal project names, no internal process vocabulary leaking into public docs.                                                                     |
| Versionless-specifier scan     | No bare pinnable `jsr:@netscript/*` specifiers; documented install/import specifiers use the `{{ releaseSpecifier }}` convention.                                                                       |
| Command/API accuracy sampling  | **Every documented command family** is sampled against the live public entrypoint; API claims are checked against `deno doc`. Documented behavior must match the shipped surface.                       |
| Template ↔ generated drift     | Where applicable, `deno task check:assets-barrel` — generated barrels/registries match their source templates; docs describing generated output are not stale.                                          |
| Nav / front-matter wiring      | New pages are wired into navigation and carry correct front-matter; no orphaned page is added.                                                                                                          |
| Prose-quality pass             | Heading structure is sound; no redundancy; one claim per callout; any shorthand/alias carries reader context. A command shown to the reader must either work out of the box or carry its setup context. |
| Cross-page contradiction check | Every changed claim is checked against the rest of the doc tree for contradiction — the changeset-scope failure mode this profile exists to catch.                                                      |

## Gate logging requirement

The audit run MUST log, per gate, in the run dir's audit artifact, a structured `## Gate log` table
with these columns:

| Column     | Content                                                                     |
| ---------- | --------------------------------------------------------------------------- |
| Gate       | The gate name (from the table above).                                       |
| Command(s) | The exact command(s) the auditor ran.                                       |
| Scope      | What the gate covered (files, command families, pages, changed lines).      |
| Result     | PASS / FAIL.                                                                |
| Findings   | Concrete findings, including specific file/line/claim references.           |
| Proceeded  | How the auditor proceeded (fixed-in-place, flagged for generator, blocked). |

**Why structured logging is required.** After a few runs, the recurring per-gate procedures captured
in these logs are meant to be **pattern-mined and promoted into `.llm/tools/docs/` automation** — the
manual command sequences become checked-in tooling, and future auditors follow proven patterns
instead of re-deriving them. Pattern-mining these logs into tools is the **intended lifecycle** of
this profile: the log table is the raw material for that automation, so keep it structured and
complete every run.

## Escalation

Two FAIL cycles on the audit → **supervisor escalation**, per the standard evaluator loop
(`escalation.md`). The auditor does not loop indefinitely; a persistently failing changeset is a
supervisor decision, not an audit retry.
