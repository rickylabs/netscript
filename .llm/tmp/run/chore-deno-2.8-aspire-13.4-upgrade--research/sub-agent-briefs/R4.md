### Part 2 — scaffold parity: source app deno.json pins from the catalog

File: `packages/cli/src/kernel/adapters/templates/app/generate-app-deno-json.ts`.

Today the generated app `deno.json` hardcodes Deno-dep literals in the `imports` block
(`fresh`, `preact`, `preact/hooks`, `@preact/signals`, `@fresh/plugin-vite`, `@tailwindcss/vite`,
`tailwindcss`, `vite`, `vite/client`). These drift from the framework's root-`deno.json` `catalog`
pins (e.g. catalog has `preact ^10.29.2`, `@preact/signals ^2.9.1`, `vite 7.2.2`).

Goal: the scaffolded app's Deno-dep versions are SOURCED from a single scaffold catalog constant
that mirrors the root `deno.json` `catalog`, not inline literals — so a catalog bump (like R3's)
propagates to new scaffolds without hand-editing this generator. Research the existing scaffold
constant layer first (`scaffold-packages.ts`, `scaffold-versions.ts`, `import-resolver.ts`, and how
`netscript-cli` describes Layer A catalog copy) and follow the established pattern; introduce a
`SCAFFOLD_VERSIONS`-style constant for these external pins if none exists. Keep the public function
signature and output shape stable (only the version strings move behind a constant).

Acceptance for Part 2: the generated `apps/<name>/deno.json` external pins equal the root-catalog
pins, and changing the catalog/constant changes the scaffold output (no literal drift remains).
