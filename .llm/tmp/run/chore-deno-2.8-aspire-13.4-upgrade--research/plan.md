## Archetype

This run is a **toolchain upgrade**, not a single package. It has no single doctrine archetype, but
its Aspire half lands almost entirely inside the **A6 — cli-tooling** package
(`packages/cli/src/kernel/{constants/scaffold,assets/aspire,templates/aspire}/`) because — per the
research's discrepancy findings — the Aspire SDK pins and the apphost are **generated by
`netscript init`**, not committed C# files. The Deno half is workspace-level config (`deno.json`,
member `deno.json`, CI). Where this run touches CLI scaffold constants it respects A6 gates; the
deeper apphost-shape realignment is **owned by Wave 6** (PR #43), not here (single-file ownership
rule, locked decision LD-8).

## Current Doctrine Verdict

N/A for the workspace-config surface. For the CLI scaffold-constant surface the live verdict is the
open `packages/cli` AP-1 "Restructure" (owned by Wave 6). This run does not restructure the CLI; it
edits `scaffold-versions.ts` (a leaf constant module) only.

## Goal

Move the workspace to **Deno 2.8.x** (adopting every program-affecting 2.8 feature, removing the
legacy it supersedes) and prepare the **Aspire 13.4.x** scaffold pins, leaving `main` green at every
merge and never emitting a broken `netscript init`. No backward-compat is retained (greenfield
alpha). This run delivers the toolchain foundation that **Phase P** (JSR alpha publish, see
`phase-p-jsr-alpha-publish-plan.md`) and **Wave 6** (CLI A6-v2) build on.

## Scope

- **Phase T (Slice 0)** — Deno 2.8 only: CI pin `v2.x → v2.8.x`; `catalog:` block + rewrite 28
  member `deno.json`; remove dead lint suppressions; normalize per-package `isolatedDeclarations`/
  `lib`; the four `--allow-slow-types` carve-outs + paired `arch-debt.md` rows; `deno ci`, `deno
  audit`, per-function coverage, `deno task --parallel` in CI.
- **Phase T prerequisite** — fold in the `packages/aspire/src/public/mod.ts` barrel
  types-vs-values split (the one red `deno publish --dry-run`; Wave-6 R-1 / slice 0.3). It is a
  publish prerequisite for Phase P, so it lands here.
- **Phase A (Slice 1 / 1b)** — Aspire 13.4 scaffold-version bump: `SCAFFOLD_VERSIONS.{ASPIRE_SDK,
  ASPIRE_HOSTING_DENO,ASPIRE_HOSTING_SQLITE}` → `13.4.x`; consolidate the duplicated
  `SCAFFOLD_COMMUNITY_TOOLKIT.VERSION`; audit `tsx`/`vscode-jsonrpc` pins; stub the `[aspire-13.5]`
  blocks in the apphost template; wire `aspire logs --search` in the nightly e2e.
- **Phase A (Slice 2)** — joint with Wave 6: `WithProcessCommand()` wired-but-flag-off (LD-4),
  `aspire doctor` skip-when-13.4 step, Aspire MCP surface documented in the `aspire` skill.

## Non-Scope

- Aspire **13.5 native Deno apphost** flip (Slice 3) — post-13.5 GA, **not a launch gate**; stubbed

- The actual `deno publish` execution — that is **Phase P** (`phase-p-jsr-alpha-publish-plan.md`),
  a separate milestone consuming this foundation.

## Hidden Scope

- **The Aspire upgrade is a CLI edit, not a C# edit** (D-1/D-2): `AppHost.csproj`/`global.json` are
  generated. Anyone treating this as a `dotnet/` change will edit the wrong files.
- `catalog:` rewrite touches **28 member `deno.json`** + the root — a wide but mechanical diff that
  must keep every member's `publish:dry-run` green (E-8).
- Re-enabling `no-node-globals`/`no-process-global` interacts with 2.8's default-on `lib.node`:
  **do not** add `"node"` to `compilerOptions.lib` (A-11) or the lint guard is defeated.
