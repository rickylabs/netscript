## Completed

### S1 — CLI embedded registry + kill URL overload

- CLI template assets now have a generated import-attribute barrel:
  `packages/cli/src/kernel/assets/embedded.generated.ts`.
- `TemplateRegistry` is content-backed and no longer carries asset URLs or fetch hydration.
- `readTemplateAsset` and `readTemplateAssetSync` accept only `TemplateKey`.
- Service, database, plugin registry, and Windows env scaffolders now use keyed template reads.
- `deno.lock` has no intended S1 churn.

### S2 — Plugin skeleton embedded content

- `@netscript/plugin/templates` now exports `PLUGIN_SKELETON_TEMPLATE_CONTENT`.
- `packages/plugin/src/kernel/assets/embedded.generated.ts` is generated by
  `deno task gen:assets-barrel`.
- CLI plugin scaffold consumes embedded skeleton content and no longer needs `templateRoot`.
- `public-command-dependencies.ts` no longer calls `import.meta.resolve('@netscript/plugin')`; public
  `netscript --help` exits 0.
- `deno.lock` has no intended S2 churn.

### S3 — Fresh UI embedded registry

- `@netscript/fresh-ui/registry` now exports `freshUiRegistryManifest` and
  `FRESH_UI_REGISTRY_CONTENT`.
- `packages/fresh-ui/registry.generated.ts` is generated by `deno task gen:assets-barrel` with one
  static import-attribute text import per registry source file.
- CLI UI installation consumes the embedded fresh-ui manifest/content by default; `--registry-root`
  remains the explicit filesystem override path.
- The old default fresh-ui filesystem root (`fromFileUrl(new URL(...fresh-ui...))`) is removed from
  the JSR-prod path.
- `renderTemplateAssetSync` now uses the existing template renderer so embedded template reads keep
  pipe support (`camelCase`, `pascalCase`, etc.).
- `deno.lock` has no intended S3 churn.
